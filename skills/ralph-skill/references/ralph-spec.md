Referenced by ralph-skill (apply/audit modes) before producing any loop prompts.

## §1. Normative Conventions

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in BCP 14 (RFC 2119, RFC 8174) when, and only when, they appear in all capitals, as shown here.

## §2. System-Level Conventions

**R.2.1** The main agent context MUST act as a scheduler only — it MUST NOT perform file-reading, code search, or build operations directly; those MUST be delegated to subagents.

**R.2.2** Each loop iteration MUST start from a clean context window. State persistence between iterations MUST occur exclusively through files on disk (`IMPLEMENTATION_PLAN.md`, `AGENTS.md`, `specs/*`). A degraded, long-running context MUST NOT be used for security-sensitive reviews (from Spec-B B.6.3).

**R.2.3** Every loop iteration MUST load the same deterministic file set: `PROMPT.md` (the active prompt file) and `AGENTS.md`. No other files MUST be pre-loaded into the main context.

**R.2.4** The first ~5,000 tokens of each loop context SHOULD be allocated to specs files to guarantee specs are in the "smart zone" of context utilization.

**R.2.5** `AGENTS.md` MUST remain operational-only (build/run/validate commands and codebase patterns). Status updates, progress notes, and planning content MUST NOT appear in `AGENTS.md`; they belong in `IMPLEMENTATION_PLAN.md`.

**R.2.6** `AGENTS.md` SHOULD be kept under 60 lines. See §8 Known Limitations for the documented empirical-vs-theoretical disagreement on this limit.

**R.2.7** The loop script MUST use either `--dangerously-skip-permissions` or `--auto` (auto mode, Team-plan research preview as of March 2026) to enable autonomous tool-call execution. Operators MUST run the loop inside a sandbox environment with minimum-viable credentials and network access. When credentials or SSH keys are reachable from the sandbox, `--auto` SHOULD be preferred over `--dangerously-skip-permissions` because auto mode's Sonnet 4.6 classifier blocks dangerous tool calls while approving safe ones.

**R.2.8** The active prompt mode MUST be selected by swapping the prompt file (`PROMPT_plan.md` vs `PROMPT_build.md`), not by runtime filtering within a single prompt.

**R.2.9** When multiple independent features require simultaneous implementation, the operator SHOULD use Git worktrees — one worktree per feature — rather than attempting to implement features in parallel within a single worktree. Each worktree MUST have its own `IMPLEMENTATION_PLAN.md`. (Sourced from Spec-B B.5.2.)

**R.2.10** For critical code reviews within the build loop — especially security-sensitive changes — the agent SHOULD invoke an Opus subagent with an explicit "Staff Engineer" or adversarial review persona, rather than self-reviewing within the same reasoning context. (Sourced from Spec-B B.6.4.)

## §3. Aspect 1 — Explore

### §3.1 Normative Statements

**R.3.1** The agent MUST study `specs/*` at the start of every loop iteration using parallel subagents before taking any planning or implementation action.

**R.3.2** Each spec file MUST cover exactly one topic of concern. A topic passes scope if it can be described in one sentence without conjoining unrelated capabilities with "and".

**R.3.3** Spec content MUST express behavioral outcomes only (observable results, measurable criteria). Implementation details (algorithms, class names, library choices) MUST NOT appear in spec files.

**R.3.4** The agent MUST study `src/lib/*` with parallel subagents to understand shared utilities and components before planning or implementing.

**R.3.5** The agent MUST NOT assume functionality is missing; it MUST confirm via code search using subagents before treating a capability as absent.

**R.3.6** Specs derived from research documents MUST carry a `## Source Attribution` section. Specs with research gaps or single-source claims MUST carry a `**[LOW CONFIDENCE — research gap]**` annotation. (Sourced from Spec-C C.7.1, C.7.5.)

### §3.2 Acceptance Criteria (Gherkin)

```gherkin
# anchor: R.3.5
Feature: No False Absence Assumption

  Scenario: Agent avoids duplicating existing implementation
    Given an implementation plan contains a task for feature X
    When the agent selects that task
    Then the agent spawns Sonnet subagents to search src/* for feature X
    And the agent only implements X if search confirms absence
    And the agent documents the search result in IMPLEMENTATION_PLAN.md
```

```gherkin
# anchor: R.3.6
Feature: Research-Sourced Spec Attribution

  Scenario: Spec derived from research carries attribution
    Given a spec file was generated from a research document
    When the spec is reviewed before planning
    Then the spec contains a Source Attribution section
    And any research-gap claims carry a LOW CONFIDENCE annotation
    And the planning agent treats LOW CONFIDENCE specs as lower priority
```

### §3.3 Rationale

The parallel-subagents-for-reading pattern preserves main-context token budget. The "don't assume not implemented" constraint prevents the most common Ralph failure mode: duplicating code that already exists under a different name or location. Behavioral-only specs maintain the "Let Ralph Ralph" principle. Source attribution on research-derived specs is essential because research documents may have gaps or single-source claims (validated by Spec-C C.7.5); without attribution, low-quality specs are indistinguishable from well-grounded ones at planning time.

## §4. Aspect 2 — Plan

### §4.1 Normative Statements

**R.4.1** PLANNING mode MUST produce or update `IMPLEMENTATION_PLAN.md` as its sole output. It MUST NOT implement any code, create commits, or modify `src/*`.

**R.4.2** The planning agent MUST use an Opus subagent for gap analysis synthesis and task prioritization. Sonnet subagents MAY be used for code reading; Opus MUST be used for prioritization reasoning.

**R.4.3** `IMPLEMENTATION_PLAN.md` MUST be a prioritized bullet-point list, sorted by implementation priority, with incomplete items listed before complete items.

**R.4.4** The planning agent MUST search for TODOs, minimal implementations, placeholders, skipped/flaky tests, and inconsistent patterns when building the gap analysis.

**R.4.5** `IMPLEMENTATION_PLAN.md` MAY be deleted and regenerated at any time. The plan is disposable; the specs and code are not.

**R.4.6** If spec files are missing for topics the planning agent discovers, the planning agent SHOULD author new spec files in `specs/FILENAME.md` before adding tasks to the plan.

**R.4.7** LOW CONFIDENCE specs (R.3.6) SHOULD be treated as lower-priority candidates in the plan until validated by the user.

### §4.2 Acceptance Criteria (Gherkin)

```gherkin
# anchor: R.4.1
Feature: Plan-Only Output in Planning Mode

  Scenario: Planning mode produces no code changes
    Given loop.sh is invoked with the "plan" argument
    When the planning loop completes
    Then IMPLEMENTATION_PLAN.md has been created or updated
    And no files in src/* have been modified
    And no git commits have been created
```

```gherkin
# anchor: R.4.7
Feature: LOW CONFIDENCE Spec Deprioritization

  Scenario: Planner defers low-confidence spec tasks
    Given IMPLEMENTATION_PLAN.md is being generated
    And specs/*.md contains a spec with a LOW CONFIDENCE annotation
    When the Opus subagent prioritizes tasks
    Then tasks derived from the LOW CONFIDENCE spec appear lower in the plan
    And the plan notes that human validation is recommended before implementing those tasks
```

### §4.3 Rationale

Strict separation between planning and building prevents the most expensive failure mode: the agent implementing the wrong thing at scale. Using Opus for synthesis is justified by the complexity of gap analysis across large codebases. LOW CONFIDENCE spec deprioritization ensures that research gaps do not silently drive significant implementation effort before the user has a chance to validate the underlying assumptions.

## §5. Aspect 3 — Implement

### §5.1 Normative Statements

**R.5.1** The building agent MUST select exactly one task from `IMPLEMENTATION_PLAN.md` per loop iteration.

**R.5.2** The building agent MUST complete its selected task, commit, and exit before the bash loop restarts with a fresh context window.

**R.5.3** The agent MAY use up to the configured maximum parallel Sonnet subagents for file reads and code searches. It MUST use only 1 subagent for build and test execution (backpressure control).

**R.5.4** The agent MUST use Opus subagents for complex reasoning tasks (debugging, architectural decisions). Sonnet subagents MUST be used for reads and searches.

**R.5.5** Implementations MUST be complete. Placeholders and stubs MUST NOT be committed; they waste subsequent loop iterations re-implementing the same work.

**R.5.6** `src/lib/` MUST be treated as the project's standard library. The agent MUST prefer consolidated implementations in `src/lib/` over ad-hoc copies in feature directories.

### §5.2 Acceptance Criteria (Gherkin)

```gherkin
# anchor: R.5.1
Feature: One Task Per Iteration

  Scenario: Agent selects and completes a single task
    Given IMPLEMENTATION_PLAN.md contains multiple incomplete tasks
    When a build loop iteration starts
    Then the agent selects the highest-priority incomplete task
    And the agent implements only that task
    And the agent commits and exits after the task is complete
    And the bash loop restarts for the next task
```

```gherkin
# anchor: R.5.3
Feature: Backpressure Subagent Isolation

  Scenario: Build/test isolation enforced
    Given a task requires both code search and test execution
    When the agent executes that task
    Then code search uses parallel Sonnet subagents
    And test execution uses exactly 1 subagent
    And no other subagents are spawned during test execution
```

### §5.3 Rationale

One task per loop is the core Ralph invariant. It enforces context isolation (Three Pillars: Context Isolation), prevents cascading failures, and makes git history readable. The backpressure constraint (1 subagent for builds/tests) prevents flaky results from parallel test execution and ensures the agent receives a single authoritative signal before committing.

## §6. Aspect 4 — Review

### §6.1 Normative Statements

**R.6.1** After implementing, the agent MUST run the validation commands specified in `AGENTS.md` (tests, typecheck, lint) before committing.

**R.6.2** If validation fails, the agent MUST fix the failures before proceeding to commit. The agent MUST NOT commit code that fails its own validation suite. (Three Pillars: Incremental Verification.)

**R.6.3** If tests unrelated to the current task fail, the agent MUST resolve them as part of the current increment, or document them in `IMPLEMENTATION_PLAN.md` if resolution is out of scope.

**R.6.4** `AGENTS.md` MUST contain the project-specific validation commands. The PROMPT_build.md instruction to "run tests" is generic; `AGENTS.md` is the authoritative source for the actual commands.

**R.6.5** For subjective acceptance criteria (tone, aesthetics, UX quality), the agent MAY use LLM-as-judge tests with binary pass/fail via `src/lib/llm-review.ts` if that module exists.

**R.6.6** For security-sensitive code changes, the agent SHOULD spawn an Opus subagent with an explicit adversarial review persona (e.g., "Staff Engineer reviewing for security vulnerabilities") rather than self-reviewing. (From R.2.10, Spec-B B.6.4.)

### §6.2 Acceptance Criteria (Gherkin)

```gherkin
# anchor: R.6.2
Feature: Backpressure Gate Before Commit

  Scenario: Agent does not commit on test failure
    Given the agent has implemented a task
    When the agent runs the test suite
    And tests fail
    Then the agent does not create a git commit
    And the agent diagnoses and fixes the test failures
    And the agent re-runs the test suite
    And the agent commits only when all tests pass
```

```gherkin
# anchor: R.6.6
Feature: Staff Engineer Review for Security Changes

  Scenario: Security-sensitive code triggers adversarial review
    Given the agent has implemented a change affecting authentication or authorization
    When the agent reaches the review step
    Then the agent spawns an Opus subagent with a Staff Engineer adversarial persona
    And the subagent reviews the diff for security vulnerabilities
    And the agent addresses any findings before committing
```

### §6.3 Rationale

Backpressure (Incremental Verification pillar) is the mechanism by which Ralph achieves eventual correctness. The AGENTS.md indirection exists so that the same generic build prompt works across projects. The Staff Engineer pattern for security reviews addresses the known failure mode of self-review within a degraded or biased context — the same reasoning that Spec-B B.6.3 and B.6.4 identify for Claude Code reviews.

## §7. Aspect 5 — Validate

### §7.1 Normative Statements

**R.7.1** When all validation passes, the agent MUST execute `git add -A`, then `git commit` with a descriptive message, then `git push`.

**R.7.2** After committing, the agent MUST update `IMPLEMENTATION_PLAN.md` to mark the completed task and record any discoveries made during implementation.

**R.7.3** When `IMPLEMENTATION_PLAN.md` becomes large, the agent SHOULD periodically remove completed items using a subagent.

**R.7.4** When the agent learns something new about how to run the application, it MUST update `AGENTS.md` using a subagent, keeping the update brief and operational only.

**R.7.5** When no build or test errors exist, the agent SHOULD create a git tag. Tags SHOULD follow semver, starting at `0.0.1` if no tags exist.

**R.7.6** When the agent discovers spec inconsistencies, it MUST use an Opus subagent with `ultrathink` to update the affected spec files.

**R.7.7** Documentation authored by the agent MUST capture the why — including why tests exist and why implementation choices were made.

### §7.2 Acceptance Criteria (Gherkin)

```gherkin
# anchor: R.7.1, R.7.2
Feature: Post-Commit Plan Update

  Scenario: Plan updated atomically with commit
    Given the agent has completed a task and all tests pass
    When the agent commits
    Then git add -A is run first
    Then git commit is run with a descriptive message
    Then the agent updates IMPLEMENTATION_PLAN.md to mark the task complete
    Then the bash loop restarts with a fresh context window
```

```gherkin
# anchor: R.7.6
Feature: Spec Consistency Maintenance

  Scenario: Agent resolves spec inconsistency
    Given the agent discovers that spec-A contradicts spec-B
    When the agent processes this finding
    Then the agent spawns an Opus subagent with ultrathink
    And the Opus subagent updates the affected spec files
    And the agent documents the resolution in IMPLEMENTATION_PLAN.md
```

### §7.3 Rationale

The commit-then-update-plan sequence is critical for the loop's resilience. If the process is interrupted between commit and plan update, the worst outcome is a duplicate task in the next iteration — not lost work. Spec inconsistency resolution by Opus-with-ultrathink is justified by the reasoning cost of cross-spec contradiction analysis.

## §8. Known Limitations & Open Questions

**1. AGENTS.md length — Documented Contradiction** (sourced from Spec-B §8 and Contradictions Encountered):
- **Claims in tension**: HumanLayer empirical testing recommends ≤60 lines to prevent frontier models from ignoring rules. Anthropic documentation implies a larger but still constrained limit (~150-200 lines). The Ralph playbook recommends ~60 lines.
- **Hypothesized cause**: Vendor documentation lag vs. real-world instruction degradation observed by power users with long AGENTS.md files.
- **Evidence to resolve**: Rigorous token-limit degradation testing on current Opus models with AGENTS.md files of varying lengths.
- **Current stance**: R.2.6 uses SHOULD ≤60 lines as the recommendation. Operators with complex projects MAY go higher but SHOULD monitor for instruction degradation.

**2. Non-deterministic backpressure** — LLM-as-judge tests (R.6.5) are non-deterministic by nature. The loop achieves eventual consistency through iteration, but the number of iterations required is unbounded. This is an intentional trade-off for subjective quality criteria.

**3. Context utilization target** — The 40-60% "smart zone" rule (R.2.4) is empirically derived, not formally proven. Projects with unusually large spec files may need to reduce subagent counts or split specs further.

**4. Model naming** — R.5.4 references Opus/Sonnet model tiers. Actual model strings in loop.sh (`--model`) are subject to Anthropic model naming changes. Ralph-skill SHOULD update these when generating files.

**5. Sandbox requirements** — R.2.7 requires a sandbox. The specific sandbox technology is out of scope for this spec; see `references/loop-variants.md` for guidance.

**6. Git worktrees and plan divergence** — R.2.9 recommends worktrees for parallel features, but if features share `src/lib/`, changes in one worktree may conflict with another at merge time. This is a known trade-off; operators SHOULD coordinate shared library changes manually.

**7. Context Window Reality Gap** (sourced from Research Document §Key Findings #11, Issue #50888):
Opus 4.7 has a 1M token context window on Max/Team/Enterprise plans. In practice, auto-compaction triggers at ~200K tokens — plan for ~150K usable per session rather than 1M. This directly constrains R.2.4: the "40-60% smart zone" target applies to the ~150K usable window, not the advertised 1M. For long-running loops, the fresh-context-per-iteration bash-loop variant is therefore structurally superior to the plugin variant (which accumulates tokens within one session until compaction triggers and causes state loss).

**8. Subagent MCP Tool Limitation** (Issue #34935):
Subagents do not inherit MCP tools — only the 8 built-in tools are available to subagents (Read, Write, Edit, Bash, Grep, Glob, WebSearch, WebFetch). This is a hard platform constraint, not configurable. PROMPT_build.md instructions MUST NOT direct subagents to use MCP tools; only the main orchestrator context can access them.

**9. Open**: Is there an optimal task granularity (in spec-lines per task) that maximises loop throughput? The current guidance lacks quantitative bounds.
