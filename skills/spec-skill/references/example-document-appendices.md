# Worked Example — Document-Level Appendices

When a document holds multiple specs (like the May 2026 research report covering Jules, Claude Code, and Gemini DR), the document itself can carry meta-content that doesn't fit inside any individual spec: shared conventions, recorded contradictions, change history, methodology. This file shows what those appendices look like, drawn verbatim from the source report.

These appendices are **optional**. A single-spec document doesn't need them. A multi-spec or research-derived spec usually benefits from at least Common Conventions and Contradictions.

What this example demonstrates:

- **Common Conventions** as a top-level table that captures the *shared* structure across specs without forcing each spec to repeat it
- **Contradictions** that are honest about disagreement — including identifying which contradictions are real vs. design-divergence
- **World-Change Log** that ties recent product changes to specific spec sections
- **Reflection History** as research-process transparency
- **Methodology Note** that lets a future reader weigh the spec's claims

---

## Common Conventions Across Systems

Placed before the first spec (Spec-A) when a document covers multiple systems. Captures the *intersection* — patterns that hold across all subjects without being repeated inside each individual spec.

```markdown
## Common Conventions Across Systems

The following normative conventions represent shared best practices that apply
universally across the systems specified in this document. Despite architectural
differences, the fundamental mechanics of guiding autonomous agents converge on
several core principles regarding context isolation and verification.

| Architectural Element              | System Application | Primary Objective                                                                                          |
| :--------------------------------- | :----------------- | :--------------------------------------------------------------------------------------------------------- |
| Separation of Planning & Execution | All Systems        | Prevents the agent from hallucinating architectural solutions by forcing human review before execution.    |
| Contextual Instruction Provisioning| Coding Agents      | Uses root-level markdown files for persistent context on conventions, test commands, and constraints.       |
| Explicit Verification Criteria     | All Systems        | Requires prompts to define objective, testable completion criteria before the agent concludes operation.   |
| Structured Output Delimiters       | Most Systems       | Uses XML tags or strict schema definitions to separate instructions, context, and variable inputs.         |
```

Followed by a paragraph naming the *baseline requirement* — usually a single architectural principle the document treats as foundational. In the source document:

> A baseline requirement across these tools is the mitigation of context degradation. As agents consume larger context windows, their ability to adhere to initial instructions wanes. Therefore, the industry standard has shifted toward modularizing instructions and relying on deterministic quality gates rather than purely natural language prompting.

When generating a Common Conventions block, keep the table to 3–5 rows. Anything more granular belongs inside individual specs.

## Contradictions Encountered

Placed after the last spec. Each contradiction has three parts:

```markdown
**Contradiction <N>: <Short Title>**

  - **Claims:** <What disagrees with what — name both sides plainly>
  - **Hypothesized Cause:** <Best guess at why the sources diverge>
  - **Resolution Evidence Needed:** <What would actually settle it>
```

Two real examples from the source document:

```markdown
**Contradiction 1: The Efficacy of Repository-Level Context Files**

  - **Claims:** Vendor documentation (Anthropic, Google) and extensive community
    frameworks assert that repository-level instruction files are essential
    for maintaining agent alignment. Conversely, a February 2026 ETH Zurich paper
    ("Evaluating AGENTS.md") claims that LLM-generated context files actively
    reduce task success rates by approximately 3% and increase inference costs
    by 20% by causing agents to over-explore the codebase.

  - **Hypothesized Cause:** There is a significant divergence between
    minimal, highly targeted human-curated constraint files versus the bloated,
    generalized files generated by automated initialization commands. The
    academic evaluation likely measured the latter.

  - **Resolution Evidence Needed:** Large-scale A/B testing distinguishing
    between strict, 50-line human-curated constraint files and lengthy,
    300-line LLM-generated documentation files.
```

```markdown
**Contradiction 2: Synchronous vs. Asynchronous Execution Models**

  - **Claims:** The terminal-native system operates synchronously, emphasizing
    real-time monitoring and immediate hook interventions. The cloud-based system
    deliberately enforces a remote, asynchronous virtual machine model.

  - **Hypothesized Cause:** This represents a divergence by design between
    systems, not a contradiction in empirical evidence. The terminal architecture
    prioritizes IDE-adjacent integration; the cloud architecture prioritizes
    background scaling and enterprise batch processing.

  - **Resolution Evidence Needed:** N/A (documented architectural divergence
    by design).
```

The second example is important: not every contradiction has resolution evidence to seek. When the contradiction is design-by-divergence, say so explicitly — that's the resolution.

## World-Change Log

A list of recent product or policy changes that affected the spec. Each entry names what changed, the date, and which spec/section is affected.

```markdown
## World-Change Log

  - **Gemini Deep Research API Architecture:** Upgraded significantly in April 2026.
    Preview models replaced by an advanced tier, splitting the service into
    speed-optimized and comprehensiveness-optimized variants. Native server
    connection support and inline chart visualization were added, fundamentally
    altering the prompt capabilities required.
    *(Affects Spec C).*

  - **Autonomous Continuous Integration Repair:** In February 2026, the cloud-based
    coding agent introduced an autonomous fixing loop and configurable authorship
    modes, allowing the agent to self-correct broken CI pipelines without
    human intervention.
    *(Affects Spec A).*

  - **Terminal Agent Installation Methods:** The vendor deprecated the standard
    package manager installation in favor of native operating system installers
    to better manage strict terminal permissions and telemetry data.
    *(Background context).*
```

The "Affects" annotation is what makes this useful in audit. When reading a six-month-old spec, you can scan the World-Change Log to know which sections need re-review.

## Query Expansion Log

For research-derived specs, document how the research that produced the spec was extended adversarially. The four standard axes:

- **Adjacent** — closely related topics that might surface unstated patterns
- **Opposing** — counter-evidence or critiques of the dominant view
- **Abstraction** — the same idea at a higher level of generality
- **Orthogonal** — apparently unrelated angles that turn out to interact

```markdown
| Axis        | Target System | Query Strategy                  | Novel Finding                                       | Modified Statement?       |
| :---------- | :------------ | :------------------------------ | :-------------------------------------------------- | :------------------------ |
| Adjacent    | Spec-A        | Syntax format specifications    | Discovered widespread open standard adoption        | Yes (Reinforced A.2.2)    |
| Opposing    | Spec-A        | PR failure rate analysis        | Discovered the autonomous CI Fixer loop update      | Yes (Created A.7.2)       |
| Adjacent    | Spec-B        | Subagent routing strategies     | Identified split-and-merge patterns                 | Yes (Created B.3.2)       |
| Opposing    | Spec-B        | Ignored instruction failures    | Identified context rot over 200 lines threshold     | Yes (Created B.2.2)       |
| Adjacent    | Spec-C        | Collaborative planning API flags| Discovered strict boolean requirements              | Yes (Created C.4.1)       |
| Opposing    | Spec-C        | Fake citation analysis          | Verified URL hallucination still occurs             | Yes (Created C.6.4)       |
```

Each row should answer: which axis was probed, against which spec, what was found, and whether a normative statement was added or modified as a result. Rows where "Modified Statement?" is "No" are still valuable — they document what was checked and didn't change anything, which is useful in future audit ("we already looked at X and concluded it didn't matter").

## Reflection History

Per-iteration self-assessment using a fixed five-question template. Documents the reasoning trajectory so a future reader (or the original author returning months later) can understand how the spec evolved and where the soft spots are.

The five questions:

1. Q1. What do I currently believe, and with what confidence?
2. Q2. What is the strongest evidence against this belief?
3. Q3. Where am I most likely wrong?
4. Q4. If I restarted now, what would I do differently?
5. Q5. What is the single highest-value next action?

```markdown
**Reflection 4 (Claude Code Iteration)**

Q1. I believe, with high confidence, that the terminal agent's primary failure
    mode is context bloat causing severe instruction amnesia.
Q2. The existence of native context compaction tools shows that the vendor
    acknowledges this, indicating that user error in timing the compaction is
    the real ongoing issue.
Q3. I am most likely wrong about the architectural complexity of the subagents;
    they function as isolated context windows rather than a complex,
    swarming multi-agent hive mind.
Q4. If I restarted, I would focus more heavily on the agent's XML parsing
    abilities for precise file management and extraction.
Q5. The single highest-value next action is to construct a rigorous steelman
    argument against utilizing subagents for overly simplistic coding tasks.
```

Five to eight reflections across a research project is typical. Don't pad — empty reflections ("everything is fine") signal the discipline isn't being applied.

## Open Questions / Unresolved

The document-level Open Questions section is a *superset* of the per-spec §8 sections — it captures questions that span multiple specs or affect the whole document. Each entry should specify what evidence or threshold would close the question.

```markdown
## Open Questions / Unresolved

  - **Context File Bloat Thresholds:** At what exact line count does a repository
    instruction file cross the threshold from helpful grounding to harmful
    distractor? The community suggests 200–300 lines, but definitive model-level
    attention benchmarks are lacking.

  - **Multi-Agent Interoperability:** Can a single standardized context file
    serve both a synchronous terminal agent and a remote asynchronous cloud agent
    without containing inherently contradictory workflow instructions?

  - **Absolute Hallucination Mitigation:** How can developers fully eradicate URL
    citation hallucinations in deep research systems without disabling the
    agent's ability to synthesize un-indexed proprietary documents?
```

Each question should be a real question, not a rhetorical one. The reader should be able to imagine the experiment or evidence that would answer it.

## Methodology Note

A short paragraph (rarely more than a paragraph) describing the authoring process — frameworks used, sources weighted, biases known. This lets a future reader or auditor weigh the spec's claims against how the spec was made.

```markdown
## Methodology Note

This research was conducted using the ReAct (Reason + Act + Observe) agentic
spine, overlaid with a structured framework for source triangulation. All
factual claims underwent triangulation, ensuring every normative statement was
backed by at least one primary vendor source and cross-verified by independent
or community documentation. The Contradictions Encountered log was actively
used to capture academic vs. industry friction. Adversarial Query Expansion drove
the discovery of advanced architectural features (autonomous pipeline repair,
collaborative planning API flags), preventing the research from stalling in
local semantic minima. Steelmanning ensured high-commitment normative statements
were pressure-tested against edge cases prior to inclusion.
```

The Methodology Note is the appendix that signals "this spec was made carefully and with discipline". A spec without one is not necessarily worse, but a spec *with* one is auditable in a way that one without isn't.

## When to use which appendix

| Appendix              | Use when…                                                                                                |
| :-------------------- | :------------------------------------------------------------------------------------------------------- |
| Common Conventions    | The document holds 2+ specs that share architectural patterns                                            |
| Contradictions        | The evidence base is contested OR the document covers competing systems                                   |
| World-Change Log      | The subject systems are evolving (most agentic-AI specs qualify)                                          |
| Query Expansion Log   | The spec was derived from a structured research process and you want that process auditable             |
| Reflection History    | Same as Query Expansion Log — research-derived specs benefit; small operational specs usually don't      |
| Open Questions        | There are questions worth flagging that span multiple specs                                              |
| Methodology Note      | The spec is going to outlive the author's memory of how it was made (i.e., almost always)                |

A quick operational spec for an internal tool needs none of these. A research-grade reference spec for a class of systems probably needs all of them. Most real specs sit somewhere in between — pick what carries weight for your readers.
