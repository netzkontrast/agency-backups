# `maintenance/schemas/`

Machine-readable schemas governing this repository's frontmatter and a few adjacent corpora.

## Canonical source of truth

| File | Role |
|---|---|
| [`header-ontology.json`](./header-ontology.json) | **Canonical.** Required L1+L2 keys per type, required body headings, per-section body schema, reciprocal-edge rules, path classification. Consumed directly by [`tools/fm/validate.py`](../../tools/fm/validate.py). The prose mirror lives at [`research/flexible-frontmatter-toolchain/output/SPEC.md`](../../research/flexible-frontmatter-toolchain/output/SPEC.md) §3–§4 and §12. When the JSON and prose drift, the JSON wins ([TASK.md §3](../../TASK.md#3-frontmatter-ontology-layered-schema-with-namespacing)). |
| [`diagnostic-explanations.json`](./diagnostic-explanations.json) | Human-readable expansion of validator anchor IDs (F.3.x, F.4.x, F.12.x, F.T.x, F.B.x). Consumed by `tools/fm/validate.py` for the `--explain` mode. |
| [`narrative-ontology/`](./narrative-ontology/) | Dramatica / NCP / novel-architect schemas. Out of scope for general operational frontmatter; loaded only by `tools/dramatica-nav/`. |

## Generated mirrors (do not edit)

The six files below are **derived artefacts** regenerated by
[`tools/fm/gen_schema_mirror.py`](../../tools/fm/gen_schema_mirror.py) on every commit.
They exist so external agents (Jules, Gemini) and downstream tools that vendor a
JSON-Schema validator can check our operational frontmatter **without cloning this
repo or importing its Python**. They MUST NOT be hand-edited; the divergence
gate in `tools/check-governance.sh` re-runs the generator and fails if the
on-disk content disagrees with `header-ontology.json`.

| File | Audience |
|---|---|
| [`l1-vault-core.schema.json`](./l1-vault-core.schema.json) | Validates the six L1 keys (`type`, `status`, `slug`, `summary`, `created`, `updated`) on any operational file. |
| [`l2-task.schema.json`](./l2-task.schema.json) | Validates `task.md` (L1 + `task_*` L2 namespace). |
| [`l2-prompt.schema.json`](./l2-prompt.schema.json) | Validates `prompt.md` (L1 + `prompt_*` L2). |
| [`l2-research.schema.json`](./l2-research.schema.json) | Validates research SPEC/readme files (L1 + `research_*` L2). |
| [`l2-skill.schema.json`](./l2-skill.schema.json) | Validates Anthropic `SKILL.md` (only `name` + `description` are required by the SDK; this repo's optional `skill_*` namespace is permitted but not required). |
| [`l2-adr.schema.json`](./l2-adr.schema.json) | Validates `decisions/<NNNN>-<slug>.md` (MADR 4.0.0 L1 + `adr_*` L2). Lifted verbatim from `header-ontology.json` `types.adr.json_schema`. |

## Regeneration

```bash
python3 tools/fm/gen_schema_mirror.py            # write mirrors
python3 tools/fm/gen_schema_mirror.py --check    # exit 1 on drift; used by check-governance.sh
```

The generator reads `header-ontology.json` and emits Draft-07 JSON-Schema
documents. It is pure-Python stdlib (no `jsonschema` dependency at generation
time — only `json`). Any change to the canonical ontology MUST be followed by a
regenerate-and-commit pass in the same commit.

## Smoke-testing with vendored `jsonschema`

External agents can validate an operational file by:

1. Parsing the YAML frontmatter (any YAML library).
2. JSON-normalising it (e.g. `json.loads(json.dumps(fm, default=str))` in Python — required because YAML auto-converts ISO-8601 dates to native date objects).
3. Validating against the appropriate mirror file:

```python
import json, yaml
from jsonschema import Draft7Validator

schema = json.loads(open("maintenance/schemas/l2-task.schema.json").read())
fm = yaml.safe_load(open("tasks/006-surface-skills-architecture/task.md").read().split("---\n",2)[1])
fm = json.loads(json.dumps(fm, default=str))   # normalise dates

errs = list(Draft7Validator(schema).iter_errors(fm))
assert not errs, errs
```

The pytest suite at [`tools/tests/fm/test_gen_schema_mirror.py`](../../tools/tests/fm/test_gen_schema_mirror.py) exercises this flow against the repo's real `task.md`, `prompt.md`, and ADR files.
