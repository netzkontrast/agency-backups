## Critical Findings

### R-1 (CRITICAL) — Missing `readme.md` in Task 052 Folder

**Location:** `tasks/052-deepwiki-integration-artifact/` (directory).

**Symptom.** `ls tasks/052-deepwiki-integration-artifact/` returns only
`task.md`. No `readme.md` exists. The task folder was created with a single
file.

**Rule violated.**

- TASK.md §2: "Every Task MUST live in a dedicated subfolder … [containing]
  `readme.md` # Directory index (per FOLDERS.md)."
- FOLDERS.md §3: "EVERY folder MUST contain a `readme.md`" listing every
  file/subfolder via relative Markdown links.
- TASK.md §7 row §7.7: The `tools/lint-structure.py` linter checks for a
  missing `readme.md` in every task folder and emits an ERROR when absent.

**Impact.** Any agent entering this task folder will find no folder index.
Per AGENTS.md AG.1.1, agents read `summary` before opening the body file —
but without a `readme.md` there is no folder-level entry point. The §7.7
linter will emit an ERROR that blocks any governance-gated commit that
touches this folder.

**Required fix.** Create `tasks/052-deepwiki-integration-artifact/readme.md`
with L1 frontmatter (`type: readme`, `status: active`, `slug:
052-deepwiki-integration-artifact`) and a `## Contents` section linking to
`task.md`. Per AGENTS.md AG.AL.1 (Assumption Log rule), the file MUST
include an `## Assumptions Log` section with at least one substantive entry
or the literal line `(none)`.

### R-2 (CRITICAL) — `## Acceptance Criteria` Instead of `## Todo`

**Location:** `tasks/052-deepwiki-integration-artifact/task.md`, final
section (lines 168–177 in the new file).

**Symptom.** The checklist section carrying the `- [ ]` items is titled
`## Acceptance Criteria`. TASK.md §5 mandates the heading `## Todo`.

**Rule violated.** TASK.md §5 section 5: "## Todo — GitHub-style checklist
(- [ ] / - [x]). The Task is `done` only when every box is checked."

**Impact.** The `tools/lint-linkage.py` §7.6 todo-completion check targets
the canonical `## Todo` heading to count unchecked items. With the heading
renamed, the linter will report zero todo items at closure, and Task 052
could be set to `task_status: done` with all eight acceptance criteria still
unchecked — silently, with no linter ERROR. The governance gate that
prevents premature closure breaks.

**Secondary impact.** TASK.md §5 specifies the required sections _in order_:
Goal → Plan → Todo → Links. The non-standard heading breaks the canonical
section order that coherence tools and human reviewers rely on.

**Required fix.** Rename `## Acceptance Criteria` to `## Todo`. The
checklist items are well-written; only the heading needs to change.

**Location:** `tasks/052-deepwiki-integration-artifact/task.md`.

**Symptom.** TASK.md §5 section 6 requires a `## Links` section: "Relative

**Impact.** An agent executing Task 052 in a future session must discover
the input artifact path (`research/gemini/deepwiki-rendering-conventions-
agentic-workflows/result.md`) by reading the body prose rather than the
standard `## Links` anchor. This is a navigability debt that compounds when
the task is later superseded or audited.

**SHOULD fix.** Add a `## Links` section citing at minimum:
- `prompts/deepwiki-rendering-conventions-agentic-workflows/prompt.md`
  (linked prompt)
- `research/gemini/deepwiki-rendering-conventions-agentic-workflows/result.md`
  (input research artifact)
- `tasks/051-deepwiki-rendering-conventions-agentic-workflows/task.md`
  (blocker task)

### R-4 (MODERATE) — Non-Standard `## Plan` Heading

**Location:** `tasks/052-deepwiki-integration-artifact/task.md`, section
beginning at "## Two-Phase Execution Plan".

**Symptom.** TASK.md §5 section 4 specifies the heading `## Plan`. The
task uses `## Two-Phase Execution Plan` instead.

**Impact.** Tooling that parses section headings by canonical name will not
recognise this section as the plan. While there is no current linter that
enforces the exact heading string (the §7.0 mapping has no check for §7.2
body schema by default), future tooling added by Task 019 (`tools/fm/
validate.py --check-body`, F.B.1 section shape mismatch) WILL enforce it.
Introducing a non-standard heading now is forward-incompatible debt.

**SHOULD fix.** Rename `## Two-Phase Execution Plan` to `## Plan`. The
two-phase structure is already clearly expressed by the nested `### Phase 1`
and `### Phase 2` subheadings inside the section.

### R-5 (MODERATE) — Completed Research Artifact in `task_affects_paths`

**Location:** `tasks/052-deepwiki-integration-artifact/task.md` frontmatter,
`task_affects_paths` list, entry:
`research/gemini/deepwiki-rendering-conventions-agentic-workflows/result.md`.

**Symptom.** `task_affects_paths` declares the paths "the Task is allowed
to modify" (TASK.md §7 §7.5). The `result.md` file at that path has
`status: completed` and `research_phase: complete` — it is a read-only
archive per RESEARCH.md §1 ("A read-mostly archive once a run is `complete`
or `archived`"). Task 052's plan reads it as input material; it has no
planned write to it.

**Impact.** An executing agent scanning `task_affects_paths` may interpret
the listing as permission to write to `result.md` — e.g., appending a note
or amending the frontmatter. This would violate the research result's
immutability. The listing also inflates the path-containment scope beyond
what the task actually modifies, weakening the audit utility of the field.

**SHOULD fix.** Remove
`research/gemini/deepwiki-rendering-conventions-agentic-workflows/result.md`
from `task_affects_paths`. Document it instead in the `## Links` section
(see R-3) as an input reference. The executing agent will consult it but
must not modify it.

## Minor Findings

### R-6 (MINOR) — Numbering Gap 047 → 051 Undocumented

**Location:** `tasks/readme.md`, entries for Tasks 047 and 051.

**Symptom.** The existing entry for Task 047 includes a parenthetical note:
"(Note: `task_id: "047"` accounts for tasks 043–046 reserved in a parallel
branch landing before this one.)" No equivalent note exists to account for
the gap between 047 and 051 (slots 048, 049, 050 skipped). The renumber
commit message (`9424491 renumber downstream analysis task from 048 to 051`)
explains the origin but that context is not surfaced in the index.

**Impact.** A future agent scanning `tasks/readme.md` will observe the 047
→ 051 jump without explanation and may interpret it as index corruption or
missing tasks, potentially triggering a spurious maintenance run.

**SHOULD fix.** Add a parenthetical note after the Task 051 entry (or as a
standalone comment) explaining: "Tasks 048–050 were reserved on the
`claude/save-research-result-ZdS9m` branch but not created; 051 is the
first deployed slot after the renumber."
