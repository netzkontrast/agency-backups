## §D. Top 5 hardest-to-reconcile claims

1. **The `/sc:spec-panel ≥ 7.0` scoring gate (SC.CMD.3.7) is invented.** The upstream skill emits a `7.2/10` value only as illustrative output; no machinery consumes it; no automation loops back to `/sc:brainstorm` on a sub-7.0 score. Adopting this as binding governance would require building the gate from scratch — and that would be a new Task, not a doc amendment. **Reject as gate; preserve as advisory framing if the maintainer wants.**

2. **`/sc:reflect --type task --analyze` (SC.CMD.6.1) creates a parallel pre-commit pipeline.** The actual close-a-task path today is: `friction-log.md` (TASK.md §4.6) → `tools/check-trust.py` (PRE_COMMIT.md §8) → `tools/check-governance.sh` (5-step incl. `tools/adr/cli.py validate`) → `/sc:createPR` (AGENTS.md CR.1). Inserting `/sc:reflect` as another mandatory checkpoint either duplicates check-trust.py's job or splits the contract across two artefacts. **Either fold `/sc:reflect`'s semantics INTO `check-trust.py` (making the friction-log richer), or reject.**

3. **`/sc:pm` always-active session-init (SC.CMD.1.1) is unenforceable.** The Gemini spec says the system "MUST prioritize AGENTS.md upon any new session initialization" via `/sc:pm`. The actual enforcement surface in this repo is `./install.sh` + `tools/check-governance.sh` per AGENTS.md SS.1/SS.2 — which checks the repo's environment, not whether `/sc:pm` ran. There is no signal in the repository telling a future agent (or a CI run) whether `/sc:pm` activated. The MUST cannot be audited. **Downgrade to SHOULD or wire a marker (e.g., `maintenance/run-log.md` entry per `/sc:pm` activation).**

4. **The 8-server ↔ 8-workspace 1:1 mapping (§0 table) is rhetorically pleasing but operationally false.** AGENTS.md does NOT bind to Serena (it binds to `tools/check-governance.sh` + `.githooks/pre-commit` + `tools/adr/cli.py synthesize`'s guarded section). FRUSTRATED.md does NOT bind to Chrome DevTools. PRE_COMMIT.md does NOT bind to Context7+Sequential — it binds to a 5-step Python pipeline and the ADR validator. Reciting the table without rewriting it sets up future agents to expect machinery that doesn't exist.

5. **`task_blocked_by` satisfaction (TASK.md §8.7) ≠ DAG completion.** Gemini §0 frames the system as a DAG; TASK.md §8.7 has a *specific* satisfaction rule — an `updated`-state Task whose successor is `done` counts as satisfied (TASK.md:401-408). A pure DAG model will reject this edge as cycle-or-incomplete. Any "DAG validator" derived from the Gemini spec MUST encode the §8.7 satisfaction-by-chain rule, or it will fire false-positives on every supersession.

## §E. Recommendations (backend-architect lens)

- **Treat the Gemini spec as a UX-and-naming proposal, not a state-machine proposal.** Its highest-leverage contribution is vocabulary (Planner/Tech-Lead split, Reflexion pattern, friction → meta-learning loop, compression discipline for research). Its lowest-leverage contributions are the binding claims about MCP-server integration and the invented 7.0 scoring gate. Lift the vocabulary; reject the bindings.

- **Resist creating a parallel pre-commit pipeline.** The repo already has a coherent gate: `tools/check-governance.sh` (5 steps) + `tools/check-trust.py` + `/sc:createPR`. Every Gemini "MUST run X before Y" claim that names an `/sc:*` command should be evaluated against: *does the existing gate produce a worse outcome without it?* If yes, wire it INTO check-governance.sh. If no, downgrade to SHOULD or REJECT. Never let an `/sc:*` command become a sibling pre-commit pipeline that sometimes runs and sometimes doesn't.

- **Strip every aspirational MCP-server citation from any merged amendment.** Serena/Sequential/Context7/Tavily/Playwright/Magic/MorphLLM/Chrome DevTools are zero-of-eight integrated. Citing them as binding ("MUST utilize Tavily") in repo-native specs sets a trap for future agents: they'll either silently ignore the rule (decay) or invent the integration (drift). Either gate: integrate the server (a real Task), then cite. Or don't cite.

- **Use the existing 8-task chain (032–039) as the integration target, not a 9th task.** The §A classification matrix shows that every Gemini § already has a host task. Authoring a new "SuperClaude integration spec" alongside the host specs would create exactly the parallel-pipeline problem this evaluation flags. Phase 5 of Task 040 should produce *patches against 032–039*, not a new top-level binding doc.
